# UI Blueprint And Implementation Plan

## 1. Цель

Интерфейс `AI Office` должен быть понятен с первого взгляда.

Пользователь должен сразу понимать:

1. что делает директор;
2. что происходит с проектом;
3. где блокировка;
4. какое следующее действие.

## 2. Навигационная модель

### Офис

- `Директор`
- `Команда`
- `Запуски`
- `Одобрения`
- `Артефакты`
- `События`

### Модули

- `CRM`
- `Звонки`
- позже `Сайты`, `Обучение`, `Документы`

### Всегда на экране

- название продукта;
- выбранный проект;
- общее состояние офиса;
- число задач в работе;
- число ожидающих одобрений;
- переход между `Офисом` и `Модулями`.

## 3. Назначение экранов

### `Директор`

- постановка цели;
- сводка проекта;
- статус директора;
- явное следующее действие.

### `Команда`

- роли и статусы агентов;
- текущие задачи;
- загрузка;
- последние назначения и запуски.

### `Запуски`

- запуски задач;
- логи;
- данные окружения;
- повторы и восстановление после сбоев.

### `Одобрения`

- ожидающие одобрения;
- история решений;
- уровень риска;
- причина одобрения или блокировки.

### `Артефакты`

- документы;
- отчеты проверки;
- краткие сводки;
- предварительный просмотр результата.

### `События`

- линия событий проекта;
- понимание, что произошло и почему.

## 4. Назначение модулей

### `CRM`

- поиск и прием данных;
- AI-сводка;
- предварительный просмотр для `amoCRM`;
- кандидаты на совпадение;
- история отправок;
- очередь ручной проверки.

### `Звонки`

- список звонков;
- сводка разговора;
- оценка качества;
- сопоставление;
- запись в `amoCRM`.

## 5. Что сейчас не так

- `CRM` выглядит как спецстраница, а не как модуль;
- нет страницы `Запуски`;
- навигация не закрепляет модель `Офис vs Модули`;
- часть UI остается англоязычной;
- директор пока не ощущается центром управления.

## 6. Первый демонстрационный срез

Нужно сначала сделать:

- общая верхняя оболочка;
- разделение `Офис / Модули`;
- `Директор`;
- `Команда`;
- `Запуски`;
- `Одобрения`;
- `Артефакты`;
- модуль `CRM`.

Этого достаточно, чтобы оценить, считывается ли новая модель продукта.

## 7. Порядок реализации

### UI-1. Навигационный каркас

- общая верхняя оболочка;
- выбор проекта;
- `Офис / Модули`;
- новая маршрутизация.

### UI-2. Усиление `Директора`

- сильная сводная панель;
- меньше шума;
- явное следующее действие.

### UI-3. Страница `Запуски`

- список запусков;
- логи;
- данные окружения;
- статусы восстановления.

### UI-4. Консолидация ядра

- единый язык;
- единая визуальная система;
- более ясная роль каждой страницы.

### UI-5. `CRM` как официальный модуль

- отдельный экранный каркас;
- место в блоке `Модули`;
- подготовка под очередь ручной проверки.

### UI-6. Модуль `Звонки` V1

- живой экран очереди звонков;
- выбранный инсайт с доказательствами и сигналами для продаж;
- понятные статусы матчинга и ручной проверки;
- переходы в `Запуски`, `Одобрения` и `Директора`.

## 8. Дизайн-правила

- не делать 3D;
- не перегружать анимацией;
- не строить декоративный “офис ради офиса”;
- не смешивать управленческие экраны и рабочие экраны модулей;
- не прятать ключевые статусы ниже первого экрана без причины.

## 9. Критерий успеха

Интерфейс удачен, если:

- разница между `Офисом` и `Модулями` видна сразу;
- `Директор` ощущается центром управления;
- `CRM` воспринимается как модуль;
- состояние проекта читается без длинных логов.

# CRM Module Spec

## 1. Цель

`CRM` — первый модуль `AI Office`.

Он нужен для контролируемого обогащения:

- читать `Tallanto`;
- принимать локально обработанные результаты по `Mango`;
- находить ученика в `amoCRM`;
- безопасно записывать результат;
- отправлять спорные кейсы в очередь ручной проверки.

## 2. Роли систем

- `Tallanto` — профиль ученика;
- `Mango` — звонки;
- `MacBook + Codex CLI` — локальная обработка и анализ;
- сервер — прием пакетов данных, сопоставление и запись;
- `amoCRM` — рабочая карточка ученика.

Правило:

- тяжелая аналитика идет локально;
- сервер не делает тяжелую LLM-обработку;
- сервер отвечает за контроль и запись.

## 3. Канонические правила

- `Контакт в amoCRM = ученик`
- `Id Tallanto = главный ключ`
- `Телефон` и `Email` в `amoCRM` часто относятся к родителю
- `phone/email` — только сигналы для поиска
- автоматическое объединение по `phone/email` запрещено

Следствие:

- брат и сестра с одним номером — нормальный кейс, а не дубль по умолчанию.

## 4. Потоки данных

### `Tallanto`

1. Локально читаем карточку ученика.
2. Нормализуем профиль.
3. Пропускаем через `Codex CLI`.
4. Отправляем подготовленный пакет данных на сервер.
5. Сервер ищет контакт по `Id Tallanto`.
6. При матче пишет результат в `amoCRM`.
7. При неоднозначности создает элемент ручной проверки.

### `Mango`

1. Локально обрабатываем звонок.
2. Получаем краткую сводку и оценку качества.
3. Ищем кандидатов по номеру телефона.
4. Уточняем матч по содержанию разговора.
5. При высокой уверенности пишем результат в `amoCRM`.
6. При неоднозначности отправляем кейс в очередь ручной проверки.

## 5. Матчинг

### `Tallanto -> amoCRM`

Порядок:

1. `Id Tallanto`
2. кандидаты по `phone/email`
3. усиление по ФИО ученика, дате рождения, ФИО родителя, классу и филиалу

### `Mango -> amoCRM`

Порядок:

1. `phone` как старт
2. уточнение по содержанию разговора
3. если кандидатов несколько, кейс уходит в ручную проверку

## 6. Правила записи

Можно обновлять:

- `Id Tallanto`
- `Филиал Tallanto`
- `Класс`
- `Баланс Tallanto`
- `Пополнено Tallanto`
- `Списано Tallanto`
- `AI-сводка`
- `AI-приоритет`
- `AI-рекомендованный следующий шаг`
- `Авто история общения`

С осторожностью:

- ФИО ученика
- ФИО родителя
- дата рождения

Не перезаписывать автоматически:

- стандартный `Телефон`
- стандартный `Email`
- ручные комментарии продаж
- ручную историю общения

## 7. Поля `amoCRM`

Обязательные:

- `Id Tallanto`
- `Филиал Tallanto`
- `Баланс Tallanto`
- `Пополнено Tallanto`
- `Списано Tallanto`
- `Авто история общения`

Рекомендуемые:

- `Последняя синхронизация Tallanto`
- `Статус матчинга`
- `AI-приоритет`
- `AI-рекомендованный следующий шаг`
- `Последняя AI-сводка`

## 8. Очередь ручной проверки

Кейс уходит в ручную проверку, если:

- `Id Tallanto` не дал матч;
- по `phone/email` найдено несколько кандидатов;
- данные по `Mango` не позволяют уверенно выбрать ученика;
- система не уверена, что это существующий ученик, а не новый ребенок семьи.

Оператор выбирает:

- существующий ученик;
- новый ученик той же семьи;
- настоящий дубль;
- недостаточно данных.

## 9. Границы V1

### Входит

- чтение и обогащение данных из `Tallanto`;
- прием результатов по `Mango`;
- безопасное сопоставление;
- запись только в контакты `amoCRM`;
- очередь ручной проверки;
- журнал аудита.

### Не входит

- полная двусторонняя синхронизация;
- автоматическое создание сделок;
- слепое объединение по `phone/email`;
- массовый импорт без ручной проверки.

## 10. Следующий шаг

- довести безопасный сопоставитель для `Tallanto` и `Mango`;
- довести review queue CRM в UI до массового операционного режима;
- привязать controlled write к реальным полям и правилам аккаунта `amoCRM`;
- только после этого запускать пилот и массовый импорт.
